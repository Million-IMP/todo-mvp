# Plan: subtasks-ui-implementation

> **Feature**: 하위 작업(Subtasks) 편집 UI 구현
> **Created**: 2026-05-14
> **Phase**: Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 투두의 하위 작업 데이터 구조는 존재하지만, 생성/수정 모달(EventModal)에서 이를 편집할 수 있는 UI가 없음 |
| **Solution** | EventModal 내에 하위 작업 리스트 관리(추가, 삭제, 토글) 섹션을 추가 |
| **Functional UX Effect** | 하나의 큰 투두를 여러 개의 작은 체크리스트로 분할하여 체계적으로 관리 가능 |
| **Core Value** | 작업의 세분화를 통한 관리 효율성 증대 및 데이터 모델의 실질적 활용 |

---

## 1. 요구사항 (Requirements)

### 1.1 하위 작업 리스트 UI
- [ ] 현재 투두에 포함된 하위 작업 목록 표시
- [ ] 각 하위 작업 옆에 완료 여부를 토글할 수 있는 체크박스 배치
- [ ] 각 하위 작업 옆에 삭제 버튼(X) 배치

### 1.2 하위 작업 추가
- [ ] 새 하위 작업을 입력할 수 있는 텍스트 필드 제공
- [ ] Enter 키 또는 '추가' 버튼 클릭 시 리스트에 즉시 반영 (낙관적 UI)

### 1.3 데이터 동기화
- [ ] 모달의 '저장' 버튼 클릭 시 부모 투두와 함께 DB에 업데이트
- [ ] 빈 제목의 하위 작업은 추가되지 않도록 방어 로직 구현

---

## 2. 구현 계획

### 2.1 UI 컴포넌트 수정
- `src/components/calendar/EventModal.tsx`: 하위 작업 섹션 추가 (Tailwind CSS 기반)

### 2.2 로직 수정
- `useState`를 사용하여 모달 내 로컬 상태로 하위 작업 리스트 관리
- `uuid` 또는 임시 ID를 사용하여 신규 하위 작업 식별

---

## 3. 성공 기준 (Success Criteria)

- [ ] SC-1: 모달에서 하위 작업을 추가하고 리스트에 즉시 나타남
- [ ] SC-2: 하위 작업을 삭제하면 리스트에서 사라짐
- [ ] SC-3: 하위 작업의 완료 상태를 토글하고 '저장' 시 DB에 반영됨
- [ ] SC-4: 저장 후 팝오버(EventPopover)에서 하위 작업 진행률이 정확히 표시됨

---

## 4. 구현 순서

1. **Phase 1 - UI 스캐폴딩**: EventModal 내 하위 작업 섹션 레이아웃 잡기
2. **Phase 2 - 상태 관리**: 하위 작업 리스트 추가/삭제/토글 로직 구현 (로컬 상태)
3. **Phase 3 - 데이터 연동**: 저장 버튼 클릭 시 API 호출 및 상태 갱신
4. **Phase 4 - 검증**: 수동 및 자동 테스트를 통한 동작 확인
